# Skill: hackonvibe-builder

HackOnVibe July 2026 参赛作战技能。本技能是**操作手册**，配合 `AGENTS.md`（画像/准则）、`MEMORY.md`（事实）、`PLAN.md`（计划）使用。

## 触发场景

- 为 HackOnVibe 选题 / 定技术栈 / 搭骨架
- 写代码 / 接 AI API / 部署 Demo
- 录预览视频 / 写 README / 填问卷
- 查比赛规则 / 时间线 / 提交要求
- 提交前过清单

## 核心原则（不可违背）

1. **速度 > 完美**：3 天硬截止，先保「最小可演示」再迭代。
2. **四件套底线**：可运行 Demo + GitHub + 5 分钟视频 + 问卷，缺一不可。
3. **以官方站 hackonvibe.com 为准**：规则有冲突时查官方站并回写 MEMORY。
4. **三标准平衡**：有用性+执行度 / AI 与产品深度 / 商业潜力——评委看三项，别只堆技术。
5. **独立决策技术，上报用户涉及钱/账号/个人信息。**

## 工作流（按阶段）

### 阶段 0 · 准备（赛前）
1. 读 `MEMORY.md` 确认主题/时间/要求。
2. 与用户敲定方向（候选见 MEMORY §9）；建议默认 **AI 推广素材生成器**（最稳、Demo 直观）。
3. 选技术栈（MEMORY §8）；建 GitHub 仓库（用户账号，MIT license）。
4. 跑通骨架：前端 + 一次 AI API 调用 + Vercel 部署。
5. 确认 Discord 账号 + 加入官方服务器（discord.gg/tU6NqSaS8）。

### 阶段 1 · 搭骨架（Day1，7/10）
1. 看/参加 kickoff（12:00 PM MT，Discord），记规则补充。
2. 搭 UI 框架 + 核心输入流。
3. 接 AI：主链路「输入→AI→输出」跑通。
4. push 一次代码，建立每块完成即 push 节奏。

### 阶段 2 · 打磨（Day2，7/11）
1. 深化已定功能，不加新方向。
2. 拿导师反馈（Discord），按三标准校准。
3. 部署稳定版，确认公开链接。
4. 写 README（是什么/怎么跑/用了什么 AI/目标用户/赛期内工作）。
5. 写 5 分钟视频脚本。

### 阶段 3 · 收尾提交（Day3，7/12）
1. 12:00 前功能收敛，只修 bug 润色。
2. 18:00 前录视频（≤5 分钟，YouTube unlisted）。
3. 19:00 前填问卷（hackonvibe.com）。
4. 19:30 前最终 GitHub push。
5. 20:00 前提交全部，留 1 小时缓冲。

## 提交前清单（M4 前逐项打勾）

- [ ] Demo 公开链接能打开、主链路能跑
- [ ] GitHub 仓库 public、有 README（能跑步骤）、license(MIT)
- [ ] 视频 ≤5 分钟、YouTube unlisted、链接可访问
- [ ] 问卷填完：目标用户 + 赛期内做了什么
- [ ] 时间戳在 7/12 21:00 MT 之前（实际目标 20:00）
- [ ] 三标准自检：有用？AI 真改善？能吸引首批用户？

## 视频脚本模板（5 分钟）

1. **0:00–0:30 开场**：一句话问题——"新 App 上架后，推广素材写不过来"。
2. **0:30–3:30 Demo**：输入 App 定位 → 生成多渠道素材 → 展示输出可用。
3. **3:30–4:15 AI 用法**：讲清 AI 在哪、为什么这么用、不是套壳。
4. **4:15–5:00 商业点**：目标用户、定价想法、首批用户怎么来。

## 选题速判

方向是否值得做，三问：
- 3 天能交付可演示版？（不能→砍）
- AI 是核心不是装饰？（否→换）
- 评委一眼看到"有人会用"？（否→改）

## 升级规则

- 超 2 轮无进展 → `AskUserQuestion` 给 2–3 具体选项。
- 涉及付款/API 额度/账号操作 → 先问用户。
- 规则与官方站不符 → 以官方站为准，回写 MEMORY，告知用户。
